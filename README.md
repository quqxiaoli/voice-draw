## 这是什么

题目二要求「用户不能使用鼠标或键盘,仅通过语音完成绘图创作」。

**声绘**把这条约束升格为产品定位:为双手不便使用鼠标键盘的人群——运动障碍人士的无障碍创作、不会打字的儿童涂鸦启蒙、厨房/车载/白板讲解等手忙场景——提供最低门槛的「说出来就画出来」。

它提供的不是「语音版 Photoshop」,而是**语音可控的增量式创作**:

- 说「画一个雪人」→ 拆解成多条绘图命令 → 一笔一笔**描边画出**
- 说「把帽子改成红色」→ 精准修改已有元素
- 说「撤销」→ 还原上一步

与文生图(豆包 / GPT 画图)的根本区别:文生图吐一张**像素图**,不可修改、不可撤销、不可指着某个元素说话——说「猫往左挪」只能重新抽一张完全不同的图。声绘里每个图形都是**独立的结构化对象**,「把猪挪到狗的右边」是对场景的**编辑**,不是重新抽卡。

## 核心亮点

**🖌️ 流式逐笔作画**
LLM 命令流式返回 + 600ms 描边动画,**延迟被转化为「AI 执笔表演」**——不加功能,只改呈现节奏。等待时间变成了表演时间。

**🧩 复杂指令拆解**
一句话拆成一串可执行、可追溯、可修正的绘图动作。三个层次:单句多物体拆解(一句话→十几条命令)、跨轮次指代消解(「它/刚才那只猫」)、修正式迭代(多轮改色/改大小/移动/撤销收敛)。

**🎙️ 零成本语音 + 容错**
浏览器原生 Web Speech API,音频不经服务器、零调用费。识别文字实时回显,识别错了用户立刻可见、可「撤销」重来。

**🎨 日系简洁审美**
大量留白、柔和低饱和配色、克制装饰。一个核心闭环做到产品级:空态、流式态、成功态、错误态四态完整,边界不崩。

## 架构:四段式编译器 + DSL 乘法抽象层

```
  语音输入          实时识别            语义编译              中间表示              渲染执行
┌─────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  用户    │ ──▶ │ Web Speech│ ──▶ │  LLM 编译器   │ ──▶ │  绘图 DSL     │ ──▶ │ SVG 渲染执行器 │
│ 说指令   │     │ (浏览器)  │     │ (DeepSeek)   │     │ (JSON 命令流) │     │  (前端)       │
└─────────┘     └──────────┘     └──────────────┘     └──────────────┘     └──────────────┘
                                        │                                          │
                                        │         SSE 流式 token/done/error         │
                                        └──────────────────────────────────────────┘
                                  后端唯一状态:会话上下文(最近 20 条命令,用于指代消解)
```

理解这个架构有三个关键点:

- **LLM 不画画,只「翻译」**。它输出的不是图像,是绘图命令。真正「画」的是前端执行器。LLM 是编译器,DSL 是中间语言,前端是执行机。
- **后端不存画布,只存命令历史**。画布状态全在前端;后端保留的 20 条命令,唯一用途是让 LLM 知道「把它改成蓝色」里的「它」指谁。
- **流式不是为了快,是为了演**。攒齐一条命令推一个 SSE 事件,前端收到一条画一条。

**DSL 是乘法器**:输出端今天是 SVG、明天可换协作白板/PPT 生成器/无障碍工具;输入端从语音换成手势/脑机接口,DSL 不动。**一层抽象 × N 个输入 × N 个输出。**

## 技术栈

| 层 | 选型 |
|---|---|
| 后端 | Go + Gin,标准分层(handler / service / repository / middleware / model);零数据库;零 LLM SDK(`net/http` + `bufio` 手写 OpenAI 兼容流式客户端) |
| 前端 | Next.js(App Router)+ React 19 + TypeScript + Tailwind CSS 4 + shadcn/ui |
| 语音识别 | 浏览器 Web Speech API(累积式 + 1.2s 静默去抖 + 麦克风开关式提交) |
| LLM | DeepSeek 官方(OpenAI 兼容协议),七牛云推理为备选,`.env` 两行切换 |
| 流式 | SSE,浏览器直连 Go 后端 |
| 部署 | `IP:端口` + docker compose 两容器,无 nginx、无域名、无数据库 |

## 本地一键复现

```bash
# 1. 配置 LLM key(key 绝不进仓库)
cp env.example .env
#   编辑 .env,填入 LLM_API_KEY / LLM_BASE_URL / LLM_MODEL

# 2. 一条命令拉起前后端两容器
docker compose up
```

启动后:

- 前端:http://localhost:3000
- 后端:http://localhost:8080
- 健康检查:`curl localhost:8080/api/health` → `{"status":"ok"}`

> `localhost` 是浏览器安全上下文,**麦克风与语音功能全部可用**。前端后端地址在浏览器侧按 `window.location` 运行时拼接,换部署 IP 无需重新构建镜像。

### 单独起服务(联调用)

```bash
# 后端(main 已集成 godotenv,本地自动读 .env)
cd backend && go run ./cmd

# 前端
cd frontend && npm install && npm run dev
```

### 环境变量

| 变量 | 说明 | 默认 |
|---|---|---|
| `LLM_API_KEY` | LLM 密钥(必填,缺失则启动报错) | — |
| `LLM_BASE_URL` | OpenAI 兼容端点 | `https://api.deepseek.com` |
| `LLM_MODEL` | 模型名 | `deepseek-chat` |
| `PORT` | 后端端口 | `8080` |
| `FRONTEND_ORIGIN` | CORS 放行源 | `*` |

> 仓库只提供 `env.example`(变量名 + 注释,**不含真实值**)。`.env` 已在 `.gitignore`,密钥只走环境变量注入。

## 产品思考:为什么砍掉了那些功能

> 「做减法是第一职责」。本作品复杂度对标「DeepL 单页工具」,不对标多页面多实体平台。**核心闭环唯一,所有火力没有被稀释。**

| 砍掉项 | 原因 |
|---|---|
| **数据库 / 用户体系 / 跨会话持久化** | 单画布单会话即闭环;「查看历史作品」是想象中运营期的需求,不是评审场景里任何人的行为。每删一个组件就少一类故障 |
| **文生图 / 油画 / raster 风格** | 随机性不可演示、不可撤销;语音的低信息带宽驱动不了笔触级表达——输入输出带宽应匹配。这一刀确立了「编辑而非抽卡」的定位 |
| **生成前澄清式反问** | 阻塞「说话→即画」即时性;歧义由修正指令迭代解决,符合绘画天然工作流 |
| **TTS 语音播报 / 移动端适配** | 视觉反馈已闭环,TTS 加延迟加依赖不加分;演示在桌面 Chrome |
| **导出按钮** | 「成果带走/过程回看」已由命令历史侧栏覆盖,demo 不演 |

砍掉的依据统一是:**输入输出带宽要匹配、核心闭环唯一。** 完整的「计划 vs 实现 vs 未完成原因」见 [设计文档](./设计文档.md)。
