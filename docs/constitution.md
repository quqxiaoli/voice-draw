# Constitution · 项目章法与代码规范

> 这份文件是所有 AI 协作的唯一规范源。任何 AI(CodeWhale/DeepSeek、Gemini、Claude)
> 写代码或改代码前,必须先读本文件,输出必须遵守这里的约束。
>
> **用法**:把本文件放在 prompt 固定前缀位置(吃 DeepSeek 缓存省钱),
> 或放进项目 `/docs/constitution.md` 让终端 agent 自动读。
>
> **分两层**:
> - **第一层(雷打不动)**:命名、错误处理、密钥、PR/commit、AI 硬约束 —— 议题无关,定稿不改。
> - **第二层(技术栈相关)**:Go/Next.js 约定 —— 标了 ⚠️ 的是议题来了可能要调的。

---

## 第一层 · 通用规范(议题无关,不变)

### 1. AI 写码硬约束(最重要,违反即返工)

- **不准引入文件里没声明的依赖/框架**。要用新库必须先在 `/docs/decisions.md` 记一笔(库名、为什么、替代方案),否则不许加。
- **不准抄整段网上代码**。比赛规则:代码重复率 ≥50% 直接拉黑。所有代码要能解释清楚为什么这么写。复用第三方代码必须在 PR 描述注明来源。
- **复用本人赛前模板/过去代码,必须在 PR 描述中声明来源**(比赛规则:复用过去代码片段未在 PR 描述注明来源 = 无效作品)。在 decisions.md 登记之外,PR 描述里必须再写一遍。
- **不准一次改多个不相关的东西**。一个改动只解决一个问题(对应下面 PR 规范)。
- **不准擅自改已确认的接口契约**。`api-contract.md` 是前后端共同依据,要改先在对话里说,人确认后才改。
- **改文件前先读文件**。基于现有代码上下文工作,不瞎覆盖。
- **写完自己说清楚改了什么**:每次产出附一句话说明(将直接用作 commit message / PR 描述)。

**「不确定怎么办」分级处理**(按代码重要性区别对待,平衡速度与质量):

- **A 级 · 核心代码**(核心业务闭环、并发/状态/数据一致性、前后端契约实现处):
  不确定就**停下来问**,不要猜着往下写。这里猜错的返工和连锁错误,比问一句贵得多。
- **B 级 · 样板代码**(CRUD、表单、列表、配置、Docker/CI、纯样式组件):
  按本规范**大胆写下去**,把不确定的点在产出说明里列一句「以下做了假设:……」,
  事后由人或 Claude review 时一并核。不要为了样板代码频繁打断节奏。
- **C 级 · 边角**(脚手架、改动极小处):直接写,跑通即可。
- 判断不了属于哪级时,**当成 A 级处理**(宁可问,不可错)。

### 2. 密钥与敏感信息(硬约束,违反即重大事故)

> 仓库截止后会公开,任何泄漏的 key 都会被盗刷。比赛官方明文警告过此事。

- **API key 只走环境变量**,通过 `.env` 注入。`.env` 必须在 `.gitignore` 里。
- 仓库里只放 `.env.example`,列出变量名 + 注释,**不含真实值**。
- **任何 AI 不准把 key、token、密码写进**:代码、文档、commit message、PR 描述、日志输出、测试文件。包括"示例代码"。
- 写到需要 key 的地方,一律用 `os.Getenv("XXX_API_KEY")` / `process.env.XXX`,缺失时启动报错并提示去配 `.env`。
- 仓库全程公开:**每次 push 前确认本次改动无泄漏**;万一泄漏,第一动作是去服务商**重置该 key**,然后再清理历史(重置优先于清理,泄漏即视为已被读取)。

### 3. 仓库生命周期(对齐比赛规则)

- 仓库**开题后创建**,不准提前建。
- 仓库**自创建起公开(public)**,提交仓库地址时需可访问。
- **推论:密钥纪律从第一个 commit 起就是生死线**——全程公开意味着任何泄漏即时暴露,不存在"转公开前清理"的窗口。
- 开题后第一个 PR:导入赛前模板(/docs、脚手架、compose),PR 描述声明"复用自本人赛前准备的模板,原创"。
- 部署与 API 余额**保持存活至评审结果公布**,不准赛完即停机。

### 4. 命名规范

- 文件名:小写 + 连字符或下划线,跟所在语言惯例(Go 用 `snake_case.go`,前端组件用 `PascalCase.tsx`)。
- 变量/函数:语义清晰,不用 `data` `tmp` `res` 这种无信息名,除非作用域极小。
- 常量/枚举:集中定义,不散落魔法值(magic number / magic string)。
- 缩写统一:同一个概念全项目用同一个词(别一会 `userId` 一会 `uid`)。

### 5. 错误处理(全局原则)

- **永不向客户端暴露内部错误细节**(堆栈、SQL、内部路径)。对外给用户友好信息 + 错误码,内部记完整日志。
- **错误要分类**:业务错误(用户输入问题,4xx)vs 系统错误(服务自身问题,5xx),区别对待。
- **统一错误响应结构**:失败时用同一套错误 JSON 结构(见第二层),用 HTTP 状态码表达结果类别(4xx/5xx),不要"永远返 200 靠 body 判断"。
- **不吞错误**:不准 `catch` 了啥也不做。要么处理,要么向上抛,要么记日志。

### 6. 日志

- 关键路径(请求入口、外部调用、错误发生处)要有日志。
- 日志带上下文(请求 ID、关键参数),但**不准记敏感信息**(密码、token、完整 key)。
- 区分级别:debug/info/warn/error,别全用一个级别。

### 6.5 测试(72h 量级,按层取舍)

> 评分看的是"代码健壮度"和 PR 的"测试方式",不是覆盖率。测试投入只放在回报点上。

- **每个 PR 必填「测试方式」**(比赛要求):写清人工验证步骤——一条 curl 命令或一串 UI 操作步骤,评委和接手的 AI 都能照做。这是底线,无例外。
- **A 级核心逻辑配表驱动测试**:纯函数/转换/解析类 service 逻辑,DS 写 Go table-driven test,成本极低、回归价值高。只测纯逻辑,不测 handler、不测 UI。
- **一个 smoke 脚本**(`scripts/smoke.sh`):按 api-contract 把每个接口 curl 一遍,断言状态码。联调时是工具,README 里是评委的复现入口,一鱼两吃。
- **不做**:UI 自动化测试、handler 单测、追覆盖率。前端验收 = 人眼 + 四态切换;边界场景 = 阶段 3 的人工清单(超长/空/特殊字符/LLM 超时)。

### 7. PR 规范(直接对齐比赛评分)

> 比赛评分「开发过程与质量」占 40%,含 PR 数量质量、commit 分布合理性。这条是拿分项。
> 比赛规则:基于 PR 添加新功能 —— 即使单人,也走 feature 分支 → PR → merge,不准直推 main。

- **一个 PR 只做一件事**:实现或修改单一功能。大功能拆成多个独立 PR 分步提交。鼓励小而细的 PR。
- **PR 标题**:一句话说明本 PR 新增/修改了什么。
- **PR 描述四件套**(比赛要求):
  1. 功能描述:这个功能干什么、怎么用
  2. 实现思路:技术选型或核心实现逻辑
  3. 测试方式:如何验证它正常工作
  4. 依赖说明:用了哪些第三方库(如有);**复用过往代码/模板的,在此声明来源**
- **PR 合并后主分支必须可运行**,评委任意时间看都能复现演示。

### 8. Commit 规范(直接对齐比赛评分)

> 比赛规则:严禁临尾突击,commit 时间戳必须落在批次窗口内,
> 仅最后一天一次性导入全部代码 = 直接判无效。

- **持续提交,小步快跑**。从开赛起就有连续 commit,不准攒到最后一次性推。
- **commit message 格式**:`<类型>: <一句话>`,类型用 `feat/fix/refactor/docs/chore/test/style`。
  - 例:`feat: 实现音频转写接口` / `fix: 修复流式输出中断未关闭连接`
- **一个 commit 一个逻辑单元**,不要把十个不相关改动塞一个 commit。
- **Git 操作权只在人手里**:所有 AI 只产代码 + 一句话总结,commit/push/PR/merge 一律由人执行(保证 author 与消息风格全程一致)。
- **逻辑单元跑通即 commit,不积攒**;一个 PR 含 1–3 个 commit 是健康态。禁用无信息消息("wip"/"fix"/"调整");某分支真改出一堆碎 commit → merge 时用 Squash and merge 兜底。
- main 开启分支保护(require PR),建仓时设置,物理杜绝直推。
- progress/tasks 的勾选更新随所属任务的 PR 一起提交;独立文档工作单独走 `docs:` PR。
- **多人组队时**(如有):各自用自己账号 commit,PR 描述清楚分工。

### 9. 文档同步(文件即真相源)

- 任何 AI 接手任务,**第一件事读 `/docs/progress.md`** 了解当前进度。
- 做了关键决策(选了某个库、定了某个架构) → 记进 `/docs/decisions.md`。
- 改了接口 → 同步更新 `/docs/api-contract.md`。
- 完成一块任务 → 在 `/docs/tasks.md` 勾掉,更新 `progress.md`。

## 第二层 · 技术栈约定

> 基于烹饪平台已有栈,按 72h 比赛做了减法。标 ⚠️ 的项议题来了确认。

### 10. Go 后端(Gin)

**项目结构**(标准分层):
```
/cmd            主入口(main.go)
/internal       核心实现(外部不可导入)
  /handler      HTTP 处理器(只管请求/响应,不写业务逻辑)
  /service      业务逻辑
  /repository   数据访问(DB/缓存)
  /middleware   中间件
  /model        数据结构 / DTO
/pkg            可复用工具(可被外部导入的)
/docs           规范与协作文档
```

**分层职责铁律**:
- handler 只做:解析请求 → 调 service → 返回响应。**不准在 handler 写业务逻辑或直接碰数据库**。
- service 写业务逻辑,不直接处理 HTTP(不碰 `gin.Context` 的 W/R)。
- repository 管数据访问,service 通过接口调它(便于替换/测试)。

**响应约定**(HTTP 语义优先):
- **用 HTTP 状态码表达结果**:2xx 成功、4xx 客户端错误(校验失败 400、未认证 401、无权限 403、找不到 404)、5xx 服务端错误。
- **成功响应**:直接返回业务数据(或包一层 `{"data": ...}` 用于分页等场景统一携带 meta)。
- **错误响应**:统一结构,只在出错时出现:
```go
type ErrorResponse struct {
    Error   string `json:"error"`             // 机器可读的错误标识,如 "INVALID_INPUT"
    Message string `json:"message"`           // 给用户看的友好信息
    Details any    `json:"details,omitempty"` // 可选,如字段级校验错误
}
```

**错误处理模式**:
- 定义业务错误类型 `BusinessError{HTTPStatus int, Code string, Message string}`,实现 `Error()`。
- handler 用 `c.Error(err)` 收集错误,**全局错误中间件**统一拦截:根据错误类型映射到对应 HTTP 状态码 + 上面的 `ErrorResponse`。
- panic 用增强版 `CustomRecovery()` 中间件兜底,返回 500 + 统一错误体,同时记完整日志,不让服务崩。
- **永不把内部错误原文(SQL、堆栈、内部路径)放进 Message 返给客户端**,那些只进日志。

**SSE 流式约定**(AI 应用核心链路,预制 demo 已验证):
- 流式接口用 SSE,事件类型只有三种:
  - `event: token` — `data: {"content": "..."}` 增量内容
  - `event: done` — `data: {}` 正常结束,前端据此收尾
  - `event: error` — `data: {"error": "...", "message": "..."}` 流中出错,结构对齐 ErrorResponse
- 后端每写一个事件立即 `Flusher.Flush()`;客户端断开(`c.Request.Context().Done()`)时停止上游调用,不空转。
- 上游 LLM 调用失败发生在流开始前 → 返回普通 HTTP 错误;发生在流中 → 发 `error` 事件后关闭流。
- 部署侧:反代(Caddy/Nginx)对该路径**关闭缓冲**,这是预制部署模板的一部分。

**数据库与缓存**:
- ⚠️ **SQLite 起步**(单文件、零运维、备份容易,72h 比赛的合理基线)。议题确实需要才升 MySQL。
- ⚠️ **Redis 不预装、随时可上**(compose 加一个 service 的成本)。启用判据只有一条:**议题里存在一个用户可感知的功能因它而明显更好**——例如 LLM 结果缓存(相同输入秒回,省钱且体验可演示)、异步长任务的进度/队列、限流保护。满足就上并记 decisions.md;"我熟所以我用"不构成理由,评委看不见的基建不加分。
- 查询设超时;repository 接口隔离,真要换库时只动一层。

**请求校验**:用 struct tag + `go-playground/validator` 做入参校验,校验失败返回清晰的 4xx 错误。

### 11. Next.js 前端

> ⚠️ 前端按议题做减法,别过度搭架子。

- **栈**:Next.js 15 (App Router) + React 19 + TypeScript + Tailwind CSS 4 + shadcn/ui。
- **SSE 流式页面默认直连 Go 后端**(Next.js route handler 透传 SSE 有缓冲风险,以赛前实测为准记入 decisions.md)。⚠️ 非流式接口可走 BFF 也可直连,议题定后统一,不混用。
- **组件**:优先用 shadcn/ui 现成组件,不重复造轮子。自定义组件放 `/components`,页面放 `/app`。
- **状态**:能用 React 内置(useState/useReducer/server component)就别引状态库。
- **类型**:接口返回类型与后端 `api-contract.md` 对齐,关键 DTO 前后端各自定义但字段名/结构逐一对照契约。
- **样式**:用 Tailwind 工具类,不写散落的自定义 CSS。设计 token 见 `ui-spec.md`,所有颜色/间距/圆角从 token 取,不准写死任意值。

### 12. UI 设计原则(差异化优势区)

> 别人比功能数量,你比「一个功能做到精致 + 有审美」。这是评分「完整度与创新性 40% + 演示表达 20%」的拿分点。

- **一个核心闭环做到产品级**:有 loading 态、有错误态、有空态、边界不崩。
- **流式反馈**:AI 类应用必有等待,要有打字机/进度等反馈,不要白屏干等。
- **审美方向**:日系简洁(留白、柔和配色、克制装饰),具体 token 见 `ui-spec.md`。
- **响应式**:至少桌面端完整,移动端不崩。

## 附录 · 议题来了要做的「填空」动作

议题发布后,按这个顺序补全文档,然后才让 AI 大规模写码:

1. 建 **public** 仓库 → 官网提交仓库地址 → 第一个 PR 导入赛前模板(描述声明复用)
2. `spec.md` —— 做什么、给谁、核心闭环、砍掉什么(做减法)
3. 回到本文件,确认第二层 ⚠️ 各项:SQLite 够不够?Redis 有没有用户可感知的用武之地?前端直连还是 BFF?
4. `api-contract.md` —— 接口契约(前后端共同依据,含流式约定)
5. `ui-spec.md` —— 页面与交互
6. `tasks.md` —— 任务拆解(每个任务对应一个 PR)
