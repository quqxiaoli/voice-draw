# 声绘 voice-draw · 设计文档

> AI 语音绘图工具 · 七牛云 × XEngineer 72 小时开发比赛(题目二)
>
> 本文件按官方要求记录:**计划支持哪些指令能力、最终实现了哪些、未完成部分的原因说明**;并附成本控制与编译器架构两章,说明本作品在"产品做减法、架构做乘法"上的取舍。

---

## 一、一句话定义与产品定位

**动嘴作画:把语音指令实时编译成结构化绘图命令,在 SVG 画布上一笔一笔画出来。**

题目二要求"用户不能使用鼠标或键盘,仅通过语音完成绘图创作"。我们没有把这条约束当成人为限制来应付,而是把它升格为产品定位——双手不便使用鼠标键盘的人群(运动障碍人士的无障碍创作、不会打字的儿童涂鸦启蒙、厨房/车载/白板讲解等手忙场景)在现有绘图工具里创作表达为零,因为这些工具的交互全部建立在手部精细操作上。**本作品提供的不是"语音版 Photoshop",而是最低门槛的"说出来就画出来"。**

与文生图(豆包 / GPT 画图)的根本区别,是本作品的差异化落点:文生图吐的是一张像素图,不可修改、不可撤销、不可指着某个元素说话——说"猫往左挪一点",它只能重新抽一张完全不同的图。本作品里每个图形都是独立的结构化对象,"把猪挪到狗的右边"是对场景的**编辑**,不是重新抽卡。**赛道不是"画得好看"(那永远打不过扩散模型),是"语音可控的增量式创作"。**

## 二、计划支持的指令能力(指令集 V1)

立项时定下指令集 V1,共 13 条,作为"计划支持"的完整清单。其中第 1–3 条是能力主轴(基础图形 → 自由图形 → 复合拆解),第 4–9 条是修正式迭代能力,第 10–13 条是辅助与收尾能力。

| # | 指令能力 | 计划说明 |
|---|---|---|
| 1 | 画基础图形 | 圆 / 矩形 / 线 / 三角,含颜色、大小、位置参数 |
| 2 | 自由图形 | 「画一只猫 / 一棵树」→ LLM 生成 SVG path |
| 3 | 复合场景拆解 | 「画一个雪人」→ 拆解为多条原语命令(评分项核心) |
| 4 | 修改颜色 | 「把它改成蓝色」 |
| 5 | 改大小 | 「大一点」 |
| 6 | 移动 | 「往右移」 |
| 7 | 删除指定图形 | 「把那只猫删掉」 |
| 8 | 撤销上一步 | 「撤销 / 不对重来」 |
| 9 | 清空画布 | 「全部清空、从头来过」 |
| 10 | 写文字 | 「写上你好世界」 |
| 11 | 填充 / 描边切换 | 切换图形的 fill / stroke 表达 |
| 12 | clarify 兜底提示 | 无法解析时给一句话提示,不阻塞主链路 |
| 13 | 导出 | 语音触发导出 PNG / SVG(按钮并存) |

立项时还设了一个分支预案:**分支 A**(LLM 生成 path 质量可用)保留第 2 条自由图形;**分支 B**(path 质量差)把第 2 条降级为预制模板库(猫 / 树 / 房子 / 太阳等 8 个),其余不变。复合拆解(第 3 条)两分支都保留,因为它是评分项核心。

## 三、最终实现情况

开发收口期对"猫 / 树 / 房子 / 五角星 / 波浪线"五项做实测,LLM 生成 path **5/5 一眼可辨**,因此锁定**分支 A(混合策略)**:几何物体(房子、太阳、旗帜)用基础原语组合,有机形体(猫、树冠、波浪、云)优先用 path 写平滑曲线。分支 B 的预制模板库未启用。

13 条指令的最终交付情况:**完整实现 11 条,部分实现 1 条,砍掉 1 条。**

| # | 指令能力 | 状态 | 实现方式 / 说明 |
|---|---|---|---|
| 1 | 画基础图形 | ✅ 完整 | `op=draw` + shape 原语。三角形以 `polygon` 的 `points` 表达,未设独立 `triangle` 原语(见下方说明) |
| 2 | 自由图形 | ✅ 完整 | 混合策略,path 为一等原语;实测 5/5 |
| 3 | 复合场景拆解 | ✅ 完整 | 一句话拆解为多条 `draw`,按"先背景后前景、先主体后细节"顺序逐条流出 |
| 4 | 修改颜色 | ✅ 完整 | `op=modify` 只输出变化属性 |
| 5 | 改大小 | ✅ 完整 | `op=modify`(属性 / transform) |
| 6 | 移动 | ✅ 完整 | `op=modify` 设 `transform: translate(...)` |
| 7 | 删除指定图形 | ✅ 完整 | `op=delete`,按语义化 id 引用 |
| 8 | 撤销上一步 | ✅ 完整 | `op=undo`,前端快照栈弹栈恢复 |
| 9 | 清空画布 | ✅ 完整 | `op=clear`,触发词收紧,只在整体清空意图下生效 |
| 10 | 写文字 | ✅ 完整 | `shape=text`,内容放 `attrs.text` |
| 11 | 填充 / 描边切换 | ◐ 部分 | **并入 `modify`**:通过修改 `fill` / `stroke` 属性间接覆盖,未设独立的"切换"op,prompt 也未专门指引该能力 |
| 12 | clarify 兜底提示 | ✅ 完整 | `op=clarify`,一句话以内(≤30 字),不反问、不解释参数 |
| 13 | 导出 PNG / SVG | ✗ 砍掉 | 见第四章未完成原因说明 |

实际交付的 DSL 操作集为 6 个 op(`draw` / `modify` / `delete` / `undo` / `clear` / `clarify`)与 8 种图形原语(`circle` / `rect` / `ellipse` / `line` / `polyline` / `polygon` / `path` / `text`)。相比立项时的设想,原语集做了两处收敛:`triangle` 并入 `polygon`(三角形是 polygon 的特例,无需独立原语),并新增了立项时未列的 `ellipse` / `polyline`(开发期发现椭圆和折线对"云、波浪、屋顶"等常见物体的表达更自然)。这说明指令集 V1 的边界不是开发前一次定死的,而是随 LLM 实测能力调整的——架构(DSL 中间层)不随实测结果变,边界随实测结果定。

### 超出计划的能力:复杂指令拆解的三个层次

第 3 条"复合场景拆解"在实现中长成了本作品最强的能力,体现为三个层次,对应官方"复杂指令的拆解与执行能力"评分项:

1. **单句多物体拆解**:一句话(如"画一片草地、一个太阳,左边再画一棵树")被编译为十几条 `draw` 命令,按绘制顺序逐条流出。
2. **跨轮次的指代消解**:"它 / 这个 / 刚才那只猫"通过后端保留的最近命令上下文,精确指向画布上已存在的元素。
3. **修正式迭代**:模糊或识别有误的指令,通过多轮"改色 / 改大小 / 移动 / 撤销"收敛成最终结果——这正是真实绘画工作流(先起笔再调整),而非问询式交互。

## 四、未完成 / 调整部分的原因说明

> 官方要求记录"未完成部分的原因"。本章逐条说明:哪些是主动砍掉的产品决策、哪些是降级处理、哪些是已知的部署期限制。每一刀都带理由,且都可在 `decisions.md` 找到对应决策记录。

### 4.1 砍掉:导出功能(指令 #13)

**状态**:未实现。立项时计划"语音触发导出 PNG / SVG,导出按钮并存"。

**原因**:零数据库决策(decisions.md D7)澄清后,"成果带走"与"过程回看"两个真实需求已被覆盖——过程回看由会话内**命令历史侧栏**承担,成果带走在 demo 场景中并非评审会经过的环节。导出按钮属纯前端工作量,但它服务的用户行为(导出收藏作品)是想象中运营期的需求,不是 72 小时评审场景里任何人的行为。在"核心闭环唯一、把全部时间换成一条链路的成品度"的原则下,它被排进打磨池且 demo 不演,最终未实现。**这不是没做完,是判断它不该占核心闭环之外的时间。**

### 4.2 降级:填充 / 描边切换(指令 #11)

**状态**:并入 `modify`,未做独立能力。

**原因**:"把猫填成红色""只要轮廓不要填充"这类需求,本质是修改图形的 `fill` / `stroke` 属性,而 `modify` op 已经能表达。为它单设一个"切换"语义,会增加 DSL 的 op 数量和 prompt 的指引篇幅,却不带来 `modify` 给不了的新能力。因此降级为 `modify` 的一种用法,prompt 未对其专门着墨。属于"用已有抽象覆盖需求,不为单一用法新增 op"的减法。

### 4.3 主动砍掉的产品范围(立项即砍,带理由)

以下能力在立项时就明确不做,构成产品的边界。砍掉的依据统一是:**输入输出带宽要匹配、核心闭环唯一、评委在 5 分钟演示里不会经过的不做。**

| 砍掉项 | 原因 |
|---|---|
| raster / 油画 / 文生图风格 | 随机性不可演示、不可撤销;语音的低信息带宽驱动不了笔触级表达——没有人能用嘴描述一幅油画的笔触。这一刀同时确立了"编辑而非抽卡"的定位 |
| 生成前澄清式反问 | 阻塞"说话→即画"的即时性,会把"响应延迟"评分项从优势做成劣势;歧义由修正指令迭代解决,符合绘画天然工作流(画家先落笔再调整,没人在落笔前接受问询) |
| 流程图 / 图表专用模式 | 指令空间发散偏题,会变成"语音版 mermaid";基础图形可拼即够 |
| 多画布 / 作品管理 / 用户体系 / 跨会话持久化 | 单画布单会话即闭环;留存需求由命令历史侧栏覆盖,跨会话管理属运营期需求 |
| 语音播报回应(TTS) | 视觉反馈(画布 + 文字回显)已闭环,TTS 加延迟加依赖不加分 |
| 移动端适配 | 演示在桌面 Chrome,README 已声明环境 |

### 4.4 已知限制:公网裸 IP 下的语音可用性

**现象**:公网部署采用 `IP:端口` 直连(不绑域名、不备案),浏览器在 `http://裸IP` 这种非"安全上下文"下会禁用麦克风,且 `crypto.randomUUID()` 属性不存在。

**处理**:
- `crypto.randomUUID` 崩溃问题,通过新增 `frontend/lib/id.ts` 的 `genId()` 解决——优先用 `crypto.randomUUID`,降级到 `Math.random` 的 RFC4122 v4 实现(session_id 与指令历史 id 不要求加密强度,降级可接受)。
- 麦克风限制属浏览器安全策略,根因是非安全上下文。权衡后**未上 HTTPS**:Let's Encrypt 不给纯 IP 签证书(需域名)、域名国内要备案、自签证书浏览器报红、收口期上 HTTPS 等于推翻"IP:端口 + docker compose 两容器、无 nginx"的部署架构并引入新故障面。代价收益不成比例。
- 替代方案:**demo 视频在本机 localhost(安全上下文)录制,语音体验 100% 完整**;公网站点作为"可访问性证明 + 让评委自己摸",核心闭环由文本输入框 fallback 覆盖。

**为什么写进文档**:评委的主要观感来源是 demo 视频,视频里语音完整则"纯语音控制"的题眼立住。把这个真实限制诚实写进"已知限制",反而说明我们清楚依赖关系——这是加分而非减分。

### 4.5 工程层面的已知遗留(不影响核心闭环)

- `scripts/smoke.sh`(按契约 curl 每个接口断言状态码)未落地,联调阶段以手工 curl 替代。
- parser 的表驱动测试未写(实现已落,属 B 级杂务,未阻塞)。

## 五、成本控制

72 小时比赛,成本控制不只是省钱,也是"把资源花在评委看得见的地方"的判断力展示。本作品的成本控制有四条主线:

**1. ASR 零成本。** 语音识别采用浏览器原生 **Web Speech API**,音频不经过服务器、浏览器直接返回识别后的文字。这一条同时省掉了三样东西:识别 API 调用费、录音上传链路、后端转写代码。云 ASR 仅作为降级备胎,实测 Web Speech 中文识别稳定,从未启用。

**2. 零数据库。** 单画布单会话,画布状态全在前端内存,后端无持久化需求。砍掉 SQLite / Redis 不是为了偷懒,而是因为"查看历史作品"是想象中运营三个月后的需求,不是评审场景里任何人的行为。**每删一个组件就少一类故障。** 为单会话工具挂数据库是简历驱动开发,资深评委看到的反应是皱眉而非加分。

**3. LLM 上下文截断。** 后端唯一的状态是会话上下文——只保留**最近 N=20 条命令**,用于"把它改成蓝色"里"它"的指代消解。条数截断既控制了每次请求的 token 成本,也避免上下文无限增长。配合 **2 小时 TTL 惰性清理**(在读写时顺带清理过期 session,5 分钟节流避免每次全表扫描),保证部署存活至评审期间内存不会单调增长。

**4. LLM 选型与协议兼容。** LLM 走 **OpenAI 兼容协议**,base_url / key 全部环境变量化,供应商切换成本是 `.env` 两行配置。最终主选 **DeepSeek 官方**(国内服务器直连可达、流式稳定、便宜),七牛云推理为备选——协议兼容使两者可随时互换,上游故障有逃生口。部署期国内源(Go 用 **goproxy.cn**、npm 用 **npmmirror**)写进 Dockerfile 跟仓库走,谁拉谁能 build,复现稳定;其中 goproxy.cn 由七牛维护,与比赛主题契合。

> 一句话总括成本控制哲学:**输入是低带宽的(语音),输出引擎高保真也是浪费——这是端到端匹配,不是妥协;评委看不见的基建不加分。**

## 六、架构设计:四段式编译器 + DSL 乘法抽象层

本作品的架构是一个**编译器结构**:

```
语音  →  ASR  →  LLM 编译器  →  绘图 DSL(JSON 命令)  →  SVG 渲染执行器
(前端)  (浏览器)   (后端)         (中间表示 / IR)            (前端)
```

理解这个架构有三个反直觉但关键的点:

**LLM 不画画,LLM 只"翻译"。** 它输出的不是图像,是绘图命令(NDJSON,每行一条 JSON 命令)。真正"画"的是前端执行器。**LLM 是编译器,DSL 是中间语言,前端是执行机。**

**后端不存画布,只存命令历史。** 画布状态全在前端。后端保留的那 20 条命令,唯一用途是让 LLM 在用户说"把它改成蓝色"时,能从历史里知道"它"指的是刚画的那个元素。

**流式不是为了快,是为了演。** 后端攒齐一条完整命令就推一个 SSE `token` 事件,前端收到一条就执行一条:图形以 600ms 描边动画"画"出来。**等待时间被转化成了表演时间——延迟被动画吃掉了,这是不加任何功能、只改呈现节奏的差异化。**

### DSL 为什么是"乘法抽象层"

中间那层 DSL(`DrawCommand`)隔离了输入端和输出端,它是一个乘法器:

- **输出端可换**:今天渲染器是 SVG 画布,明天可以是协作白板、PPT 生成器、儿童编程教具、无障碍创作工具——指令集不变,渲染端任意换。
- **输入端可换**:输入端从语音换成手势、脑机接口,DSL 也不动。

**一层抽象 × N 个输入 × N 个输出。** 这是编译器架构(前端 / IR / 后端)的经典格局,也是官方价值观里"架构做乘法的格局"的实物证据。

与之配套的工程判断,是后端状态层用 `SessionStore` 接口 + 内存 map 实现:**留好了 Redis / SQLite 的插座,替换时 service 零改动——插座比电器更能证明懂工程。** 这是以可扩展性设计而非组件堆叠来展示工程能力。

### DSL 命令结构(契约真相源)

每条命令是一个扁平结构,按 op 决定必填字段:

```json
{"op":"draw","id":"snowman_body","shape":"circle","attrs":{"cx":500,"cy":500,"r":120,"fill":"#FFFFFF","stroke":"#3D3D3D"}}
```

| op | 必填字段 | 语义 |
|---|---|---|
| `draw` | id + shape + 非空 attrs | 新增图形元素 |
| `modify` | id + 非空 attrs | 改属性(改色 / 改大小 / 移动) |
| `delete` | id | 删除指定元素 |
| `undo` | — | 撤销上一步(前端快照栈) |
| `clear` | — | 清空画布 |
| `clarify` | message | 无法解析时的一句话提示 |

`attrs` 透传 SVG 原生属性(`cx`/`cy`/`r`/`d`/`points`/`fill`/`stroke`/`transform`…),后端不枚举校验属性名——表达力留给 LLM,渲染端直接挂到 SVG 元素,新增属性零契约改动。画布逻辑坐标系为 `viewBox 0 0 1000 750`。

## 七、容错与边界处理

官方要求综合考虑"指令理解的准确性与容错性"。本作品的容错是多层的:

- **识别有误用户可见**:Web Speech 的识别文字实时回显,用户看见识别错了立刻知道为什么画错,可以说"撤销"重来。
- **LLM 输出单行坏不影响整流**:NDJSON 解析器逐行解析,某一行 JSON 非法或校验不过就丢弃并记日志,不中断整条流。
- **永远有反馈**:一条有效命令都没产出时,后端兜底发一条 `clarify`("没太听懂这条指令,试试「画一只猫」或「把它改成蓝色」?"),用户侧不会白屏。
- **部分成功不丢上下文**(decisions.md D13):流中出错或用户中途停止时,**已经画出的命令照常进会话上下文**——上下文语义是"前端画布的认知镜像",而非"成功事务日志"。否则"画一半停止再修改"会因前后端画布认知分叉而演翻。
- **流健壮性**:LLM 流空闲超时 30 秒触发 `UPSTREAM_TIMEOUT`,前端走错误态而非永久转圈;客户端断开时后端停止消费上游,不空转。
- **四态完整**:空态(引导文案 + 示例入口)、流式中(描边动画 + 停止按钮)、成功态、错误态(柔和提示条、保留输入、可重试、绝不白屏)。

一个真实的容错例子(可作 demo 素材):语音把"猪挪到狗的右边"识别成残句、画布上没有"狗" → 系统回"画布上没有狗,无法执行" → 用户重说完整指令 → 完成。**容错-修正闭环在真实语音噪声下自然发生,这就是 demo 的剧本,不用编。**

## 八、技术栈与部署

**后端**:Go + Gin,标准分层(handler / service / repository / middleware / model)。SSE 流式直连,零数据库,零第三方 LLM SDK(`net/http` + `bufio` 手写 OpenAI 兼容流式客户端)。

**前端**:Next.js(App Router)+ React 19 + TypeScript + Tailwind CSS 4 + shadcn/ui。SSE 用 `fetch` + `ReadableStream` 手解(EventSource 不支持 POST),描边动画基于 `getTotalLength` + `stroke-dasharray/dashoffset` 过渡。

**ASR**:浏览器 Web Speech API,累积式识别 + 1.2 秒静默去抖 + 麦克风开关式提交。

**LLM**:DeepSeek 官方(OpenAI 兼容协议),七牛云推理为备选,`.env` 两行切换。

**UI 基调**:日系简洁——大量留白、柔和低饱和配色、克制装饰。主题色灰调松绿 `#7C9A83`,页面底色暖白 `#FAF9F6`,圆角统一 10px,动效只做状态过渡(150ms)与描边/打字机两种。

**部署**:`IP:端口` + docker compose 两容器(前端 + 后端),无 nginx、无域名、无数据库。评委一条命令复现:`cp env.example .env`(填 LLM key)后 `docker compose up`。前端后端地址在浏览器侧按 `window.location` 运行时拼接,换部署 IP 无需重新构建镜像。
